How it works — modes & scoring

This is the Prisoner's Dilemma framed as a model of cooperative CPU scheduling. A move of Cooperate models a program that yields the CPU (POSIX sched_yield()) so others can run; Defect models a program that keeps running to maximise its own throughput. Strategies are written and scored as ordinary Prisoner's Dilemma players — the cores view is just what the payoffs represent.

The three modes

  • 1v1 — a round-robin: every strategy plays every other strategy in a separate two-player match; results aggregate into the weighted score below.
  • N-Player — a simultaneous group match where all players move at once each round (hence the dedicated group payoff models below). Also scored by the weighted score.
  • OS Simulation — a standalone benchmark scored on scheduling metrics (throughput, wait, turnaround, response, cache misses), not the weighted score.

How the score is calculated (1v1 & N-Player)

Each strategy gets three component scores, each a fraction in [0, 1]:

  • Win rate = wins / (wins + draws + losses). Each opponent (1v1) or group match (N-Player) counts as one game, not one per round; the winner is the side with the higher average points per round.
  • Points = avg_points_per_round / max_round_points, where avg_points_per_round = total_points / total_rounds_played and max_round_points is the largest single-round payoff available (e.g. 5 for the default matrix). A throughput proxy in [0, 1].
  • Cooperation = the mean of each selected mode's cooperation rate (coop_rate = cooperate_moves / moves_in_that_mode), with modes weighted equally so a long Stochastic match can't outweigh a short Standard one.

The final Weighted Score sums each fraction × its weight, scaled to 100:

Score = (win_rate·Wwin + cooperation·Wcoop + points·Wpoints) × 100

Weights are chosen when the tournament is configured and fixed for that run. Because Points use the per-round average, Win rate counts once per opponent, and Cooperation weights modes equally, match length never biases the score.

Game modes

  • Standard — a fixed number of rounds; each round's payoff added normally.
  • Discounted — round t's payoff is multiplied by discount_factort, so future rounds matter less.
  • Stochastic — after round 1 the game continues each round with probability stochastic_prob and otherwise stops, so match length varies.
  • Random Iterations — a seeded random round count (same seed ⇒ same length); scores are normalised so the random length doesn't bias results.

Selecting multiple modes runs the tournament once per mode and aggregates the scores.

N-player payoff models

  • Public Goods — each cooperator pays cost c; the pool returns benefit b·k (or non-linear b·kα) split among all n players. A cooperator nets (b·k)/n − c; a defector nets (b·k)/n (where k = number of cooperators).
  • Pairwise Matrix — each player plays the 2-player matrix against every other in the group; per-round results are aggregated (sum or average).
  • K-Cooperator Tensor — payoff depends only on the cooperator count k: a cooperator gets u_C[k], a defector gets u_D[k], read from the supplied tables.

How the OS simulation works

OS Simulation is a standalone benchmark (independent of the tournaments). Each selected strategy becomes the scheduling personality of a simulated multi-core CPU (OSSimulator), and we measure how well it schedules a workload.

  • Cores — you choose 2–8. There is no ring pairing: at each dispatch a core plays one PD move against a shared contention signal — the system is "congested" (the core sees Defect) when the ready queue is longer than the core count (len(ready) > num_cores) or a majority of the other cores are currently defecting; otherwise it sees Cooperate. Cooperate = yield the CPU; Defect = keep running.
  • Assignment — Homogeneous benchmarks each strategy alone on all N cores; Heterogeneous places different strategies on different cores and ranks the mixtures.
  • Baseline — every run also executes the plain version of the chosen scheduler (cores never yield) on the same workloads, drawn as a reference so you can see what the strategies add over the textbook policy.
  • Cooperation — each strategy's realised Cooperate rate is reported and plotted, so OS behaviour lines up with the 1v1 / N-Player cooperation metric.

The workload dataset

Each run executes against a seeded, synthetic dataset of 50 processes. Every process is drawn from the run's Seed (so a seed reproduces the exact dataset) and is defined by three things:

  • Arrival time — when it joins the ready queue (set by the profile, below).
  • CPU burst — how many ticks of CPU it needs.
  • Memory working set — a per-process footprint (log-normal, median 4 KB, σ 0.5, clamped 1–32 KB) from which a per-tick memory-access stream is generated; those addresses drive cache hits/misses.

Four profiles model distinct, well-studied workload shapes (benchmarked across all four; viewable individually or averaged via the results dropdown):

ProfileArrivalsCPU bursts
Uniformevery 15 ticks (steady)flat 30–80 (light-tailed)
Poissonexponential, mean 20 tickslog-normal (median 45, σ 0.6), clamped 10–400
Mixedevery 10 ticksbimodal: 80% short (median 20) + 20% long (median 280)
Burstybatches of 10, every 300 tickslog-normal (median 35, σ 0.55)

Why representative: real CPU service times are heavy-tailed (most jobs short, a few very long), so bursts use a clamped log-normal; arrivals use the Poisson model; and working-set sizes vary so cache pressure differs across jobs. Together the four span light/steady, random, interactive-plus-batch and spiky loads.

How the simulation runs & records metrics

The simulator advances one tick at a time. Each tick:

  1. Processes whose arrival time has passed are admitted to the ready queue.
  2. The scheduler does periodic housekeeping (MLFQ boosts, affinity load-balancing).
  3. For each core: if it is idle or its time-slice expired, the core plays its PD move against the contention signal, the scheduler selects the next job and sets its quantum; the running job executes one tick. Memory accesses hit the L1/L2/shared-L3 cache; cache stalls slow progress (work_done = 1 / (1 + stall·0.01)), decrementing the job's remaining time.
  4. On completion, the process's timing is recorded.

From the completed processes (n of them, over T total ticks) the metrics are:

turnaround = completion − arrival  |   Avg Turnaround = mean(turnaround)
waiting = max(0, turnaround − burst)  |   Avg Wait = mean(waiting)
response = first_start − arrival  |   Avg Response = mean(response)
Throughput = n / T × 1000  (jobs per 1000 ticks)
Cache Misses = (Σ per-core L1+L2 misses + shared-L3 misses) / n
Makespan = T  |   Cooperation % = Cooperate moves / total moves

Lower is better for wait/turnaround/response/cache; higher is better for throughput/cooperation. Each metric is computed per workload profile and then averaged across the four (and viewable per profile). The cache is a fixed Skylake-class hierarchy (per-core L1 32 KB + L2 512 KB, shared L3 8 MB, 64 B lines, LRU), so cache-miss numbers are always real. Reproducibility: provide a Seed to make a run (and its dataset) fully deterministic.
